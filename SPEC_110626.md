## 2. Переменные окружения (.env.local)

```env
DATABASE_URL="postgresql://..."
DIRECT_URL="postgresql://..."
JWT_SECRET="..."
```

- `DATABASE_URL` и `DIRECT_URL` обязательны для работы Prisma.
- `JWT_SECRET` используется для подписи токена аутентификации.
- В проекте нет текущей реализации Supabase Storage или Supabase Auth, поэтому переменные `NEXT_PUBLIC_SUPABASE_*` и `SUPABASE_SERVICE_ROLE_KEY` не требуются.

## 3. Схема базы данных (Prisma)

### 3.1 Роли и перечисления

```prisma
enum Role {
  SUPER_ADMIN
  ADMIN
  LAWYER
  HEAD
  EMPLOYEE
}

enum ContractStatus {
  DRAFT
  COLLECTING
  REVIEWING
  APPROVAL
  SIGNED
  ARCHIVED
}

enum DepartmentTaskStatus {
  PENDING
  IN_PROGRESS
  SUBMITTED
}

enum ApprovalAction {
  APPROVED
  RETURNED
}
```

### 3.2 Таблицы

```prisma
model Department {
  id                  String               @id @default(cuid())
  name                String               @unique
  createdAt           DateTime             @default(now())
  updatedAt           DateTime             @updatedAt

  users               User[]
  contractDepartments ContractDepartment[]

model User {
  id           String        @id @default(cuid())
  email        String        @unique
  name         String
  passwordHash String
  role         Role

  updatedAt    DateTime      @updatedAt

  department   Department?   @relation(fields: [departmentId], references: [id])
  comments     Comment[]

model Contract {
  id                  String               @id @default(cuid())

model ContractFile {
```

## 4. Бизнес-правила

### 4.1 Доступ по ролям

| Действие | SUPER_ADMIN | ADMIN | LAWYER | HEAD | EMPLOYEE |
|---|---|---|---|---|---|
| Создать пользователя | + | - | - | - | - |
| Создать отдел | + | - | - | - | - |
| Создать договор | + | - | + | - | - |
| Назначить отделы к договору | + | - | + | - | - |
| Оставить комментарий | + | + | + | + | + |
| Подтвердить сдачу отдела | + | - | - | + | - |
| Согласовать / вернуть договор | + | - | - | + | - |
| Просмотр всех договоров | + | + | + | - | - |
| Просмотр договоров своего отдела | + | + | + | + | + |
| Просмотр всех пользователей | + | - | - | - | - |
| Просмотр всех отделов | + | - | - | - | - |

### 4.2 Переходы статусов договора

```
DRAFT       → COLLECTING   юрист назначил отделы и запустил сбор информации
COLLECTING  → REVIEWING    все ContractDepartment перешли в SUBMITTED
REVIEWING   → APPROVAL     юрист отправил на согласование
APPROVAL    → COLLECTING   хотя бы один HEAD вернул договор на доработку
APPROVAL    → SIGNED       все HEAD согласовали текущую версию
SIGNED      → ARCHIVED     юрист или ADMIN переводит в архив вручную
```

- Переход COLLECTING → REVIEWING происходит автоматически, когда все `contractDepartments` имеют `status === SUBMITTED`.
- Переход APPROVAL → SIGNED происходит автоматически, когда все назначенные `HEAD` из отделов согласовали файл текущей версии.
- Возврат `RETURNED` переводит договор обратно в `COLLECTING`.

### 4.3 ActivityLog события

```
contract.created           metadata: { title, counterparty }
contract.status_changed    metadata: { from: ContractStatus, to: ContractStatus }
contract.file_uploaded     metadata: { version: Int, filename: String }
department.assigned        metadata: { departmentId, departmentName }
department.status_changed  metadata: { departmentId, from: DepartmentTaskStatus, to: DepartmentTaskStatus }
comment.added              metadata: { commentId }
approval.approved          metadata: { fileVersion: Int, comment?: String }
approval.returned          metadata: { fileVersion: Int, comment?: String }
```

### 4.4 Файлы

- В текущей реализации `ContractFile` существует, но файловая загрузка через API не реализована.
- Поле `storagePath` присутствует в модели, но использование файлового хранилища пока не подключено.

## 5. API эндпоинты

### Авторизация
```
POST /api/auth/login      body: { email, password }
POST /api/auth/logout
GET  /api/auth/me         → { data: user }
```

### Пользователи (SUPER_ADMIN)
```
GET  /api/users           → { data: User[] }
POST /api/users           body: { email, name, role, departmentId?, password }
```

### Отделы (SUPER_ADMIN)
```
GET  /api/departments     → { data: Department[] }
POST /api/departments     body: { name }
```

### Договоры
```
GET  /api/contracts
POST /api/contracts       body: { title, counterparty, object?, amount? }
GET  /api/contracts/[id]  → полная карточка договора
POST /api/contracts/[id]/departments          body: { departmentIds: string[] }
PATCH /api/contracts/[id]/departments/[deptId] body: { status: DepartmentTaskStatus }
POST /api/contracts/[id]/send-collecting
POST /api/contracts/[id]/send-approval
POST /api/contracts/[id]/approvals            body: { action: ApprovalAction, comment? }
POST /api/contracts/[id]/archive
POST /api/contracts/[id]/comments             body: { text }
```

### Формат ответов
```typescript
// Успех
{ data: T }

// Ошибка
{ error: string }
```

- HTTP статусы: 200 OK, 201 Created, 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found.

## 6. Middleware

- `middleware.ts` проверяет наличие JWT-токена в cookie `token`.
- Если нет токена и путь не `/login`, происходит редирект на `/login`.
- Если токен есть и пользователь пытается открыть `/login`, происходит редирект на `/contracts`.
- Проверка ролей выполняется внутри API route-хендлеров.

## 8. Правила для Claude Code

- Прочитай `SPEC_110626.md` перед началом работы.
- Не изменяй логику без необходимости — добавляй функциональность аккуратно.
